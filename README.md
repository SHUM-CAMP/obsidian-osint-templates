### Purpose of this project

We maintain this collection as a **shared foundation for OSINT documentation in Obsidian**. The upstream templates and example structure give us a practical starting point for linked notes, tasks, and SOP-oriented workflows. SHUM will use this collection as a **base layer for further work**, gradually expanding it with additional templates, notes, workflows, and evidence-recording structures designed specifically for our investigative, analytical, and human-rights-related tasks.

This means the current template set is not treated as a final product. It is a **foundation** on top of which SHUM can build organization-specific approaches for case work, documentation, verification, reporting, collaboration, and training, while preserving the required attribution and license obligations attached to the original materials.

## Why Obsidian for this kind of work

The upstream materials present Obsidian as a local, file-based note-taking environment that is especially useful for linking people, accounts, domains, infrastructure, organisations, evidence fragments, and analytical notes. That makes it particularly suitable for OSINT and adjacent investigative workflows.

For SHUM, this is valuable not simply because Obsidian stores notes, but because it supports:

- **structured documentation of findings**
- **local and portable knowledge storage**
- **wikilinks between entities, cases, and evidence**
- **task tracking inside notes**
- **graph-based navigation of relationships**
- **room for SOP-driven and template-driven workflows**

This is aligned with the type of work SHUM does: investigations, documentation, analytical support, counter-disinformation work, and human-rights-oriented evidence handling.

## How to use this vault

The original **START HERE** note explains the intended use of the vault and should still be treated as the practical model behind this combined document.

### Templates

The upstream templates are examples of how Obsidian can support OSINT investigation workflows. In particular, the original vault highlights reusable templates such as **[[person]]** and **[[domain or ip]]** to show how investigators can standardize the collection of recurring information types.

The intended use for SHUM is similar:

- keep the upstream templates as a reference baseline
- adapt them for real organizational workflows
- add new templates for entities, incidents, cases, sources, geospatial items, evidence containers, analytical products, and SOP checklists
- align new templates with SHUM’s own standards and documentation needs

> [!tip]
> A practical next step is to embed SHUM-specific SOP tasks directly into templates so that new notes automatically inherit the relevant checklist structure.

### Linking files

One of the strongest features highlighted in the upstream vault is the use of **Wikilinks** (`[[]]`) and tags to create relationships between notes. The **Graph View** then visualizes those connections.

For SHUM, this is useful for connecting:

- people and organisations
- incidents and evidence
- platforms and accounts
- locations and geospatial clues
- legal context and analytical notes
- tasks and case progression

### Tasks

The original vault also demonstrates how tasks can be aggregated across the workspace. That can be extended for SHUM into lightweight case-management support inside Obsidian, especially for:

- follow-up actions
- verification steps
- collection tasks
- analytical review tasks
- publication or reporting checkpoints

### SOP material

The upstream vault keeps Standard Operating Procedure material in a separate area. That approach should be preserved and expanded. For SHUM, this can become a dedicated section for:

- verification procedures
- documentation standards
- legal and evidentiary handling notes
- platform-specific collection procedures
- reporting rules
- internal methodology references

### Example notes

The original vault includes example notes intended to demonstrate capabilities. Those examples are useful as a structural reference, but SHUM’s long-term value will come from replacing or supplementing them with real, organization-specific note structures and working examples appropriate to its mission.

## Recommended adaptation approach for SHUM

To keep this collection usable and legally clean, the adaptation path should be simple:

1. **Preserve author and license credit** in the main front document and in any redistributed derivative version.
2. **Keep upstream-inspired materials identifiable** so it remains clear what originated with Micah Hoffman and what SHUM added.
3. **Add SHUM-specific templates separately**, rather than rewriting everything at once.
4. **Use this collection as a foundation**, not a constraint.
5. **Contribute back upstream** only where the improvements are general-purpose and appropriate for public sharing.
6. **Keep organization-specific operational content** in SHUM-managed notes, repositories, or internal branches where needed.
